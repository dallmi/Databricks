!2 Three Join Patterns in This Domain

!3 1. Mailing-centric (starts from tbl_email)
For "what happened to mailing X?". See !see .CrossChannelAnalytics.JoinStrategy.ImepBronzeEmailEvents.

!3 2. Person-centric (starts from tbl_hr_employee)
For "which engagement events did person X have?". Only possible via the two hub tables (tbl_email_receiver_status, tbl_analytics_link). '''SharePoint side has no equivalent person key.'''

!3 3. HR enrichment (from any TNumber-bearing table)
For "enrich my engagement fact with Region/Division". See !see .CrossChannelAnalytics.JoinStrategy.HrEnrichment.

!2 Key Observations

 * '''Two full-key hubs''': Only tbl_email_receiver_status and tbl_analytics_link carry Id + EmailId + TNumber simultaneously. These are the only tables with '''person-level engagement granularity'''.
 * '''TrackingId scope''': In Bronze only on tbl_email and tbl_event. '''Never''' on engagement tables — always fetch via tbl_email.Id = EmailId.
 * '''tbl_email_links vs tbl_analytics_link''': Don't confuse. tbl_email_links = template URL inventory (static). tbl_analytics_link = event fact table (open/click). Most common mistake.
 * '''HR join chain''': engagement-table -> tbl_hr_employee -> tbl_hr_costcenter (via ORGANIZATIONAL_UNIT). tbl_hr_user adds Town only; not needed for Region/Division.
