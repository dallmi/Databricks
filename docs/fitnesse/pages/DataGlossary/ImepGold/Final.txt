!1 Final ('''imep_gold.final''')

'''The denormalized consumption endpoint for email engagement.''' 520M rows, fully rebuilt from scratch on every run (no incrementality). Combines tbl_email + tbl_email_receiver_status + tbl_analytics_link with HR enrichment (Region, Division, etc.) — in a single table. Email engagement intentionally has '''no Silver layer'''; final '''is''' the Silver-skip endpoint.

!2 Meta
| '''Layer'''          | Gold (Consumption) — '''Tier 0 (Atomic Fact)''' |
| '''Source systems''' | iMEP Bronze (3 tables) + HR (Bronze) |
| '''Grain'''          | '''1 row per mailing × recipient × event × hour''' — events are hour-bucketed |
| '''Primary key'''    | Id |
| '''Cross-channel key''' | TrackingId (inherited via join on Mailing master) |
| '''Write pattern''' | '''Full Rebuild''' — no incrementality (Service Principal) |
| '''Rows'''           | '''~520M''', 64,196 distinct mailings |
| '''Physical storage''' | External Delta, ADLS path abfss://gold@<gold-acc>/.../final, '''no partitioning''' |

!2 Primary Joins

!3 Pattern A: Direct consumption (default for dashboards)

{{{
SELECT TrackingId, Region, Division, LogStatus, COUNT(*) AS events
FROM   imep_gold.final
WHERE  event_time >= '2025-01-01'
  AND  TrackingId IS NOT NULL
GROUP BY TrackingId, Region, Division, LogStatus
}}}

!3 Pattern B: Pack-level aggregation (dashboard grain)

{{{
SELECT array_join(slice(split(UPPER(TrackingId), '-'), 1, 2), '-') AS tracking_pack_id,
       COUNT(DISTINCT CASE WHEN LogStatus = 'Sent'  THEN TNumber END) AS sent_unique,
       COUNT(DISTINCT CASE WHEN LogStatus = 'Open'  THEN TNumber END) AS open_unique,
       COUNT(DISTINCT CASE WHEN LogStatus = 'Click' THEN TNumber END) AS click_unique
FROM   imep_gold.final
WHERE  TrackingId IS NOT NULL
  AND  event_time >= '2025-01-01'
GROUP BY 1
}}}

!2 Quality Caveats

 * '''⚠️ Zero Partitioning''' — biggest structural performance gap across all 114 tables in the system. Queries without a tight time or TrackingId filter trigger a full 520M-row scan. An iMEP-team architecture concern, not ours to fix.
 * '''⚠️ Full Rebuild''' — the table is completely destroyed and rebuilt, not incrementally merged. Long-running queries parallel to a rebuild run may encounter a momentarily inconsistent state. Refresh cadence not documented — we don't have a complete job scheduler view.
 * '''No incrementality on the Gold side''' — any change in Bronze only lands in final after the next rebuild run. For out-of-band questions about newer events, use Bronze directly.
 * '''Full scan is expensive''': 520M rows × wide schema. Always restrict queries with event_time or TrackingId filter. The Gold Full Rebuild has '''no Z-Order''' — no queries optimized for specific columns.
 * '''HR snapshot at build time''': Region/Division reflect the HR state at the time of the Full Rebuild. If an employee moves EMEA -> APAC on July 1, all older final events show the new value '''after the next Full Rebuild''' — not the historical one. For true temporal HR analyses, join from Bronze with a tbl_hr_employee snapshot.
 * '''Biggest compute hotspot in the pipeline budget''' — full rebuild of a 520M-row table. Identified as the first optimization lever, but not our concern.

'''No Silver in between''' — imep_silver exists, but only for events (invitation, eventregistration, event), '''not for email'''.

!2 Consumption Strategy

 * '''Default''': use imep_gold.final. Pre-assembled, HR-ready.
 * '''If you can't find something in final''' (e.g. a specific Bronze column): don't try to extend final, join Bronze directly.
 * '''If final gets too large''': use the Tier-3 aggregates (tbl_pbi_mailings_region, _division, tbl_pbi_kpi) — they are pre-aggregated via GROUP BY MailingId × Dimension.
