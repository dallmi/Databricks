!1 Final ('''imep_gold.final''')

'''The denormalized consumption endpoint for email engagement.''' 520M rows, fully rebuilt 2x/day via CTAS. Combines tbl_email + tbl_email_receiver_status + tbl_analytics_link with HR enrichment (Region, Division, etc.) — in a single table. Email engagement intentionally has '''no Silver layer'''; final '''is''' the Silver-skip endpoint. (Q26)

!2 Meta
| '''Layer'''          | Gold (Consumption) |
| '''Source systems''' | iMEP Bronze (3 tables) + HR (Bronze) |
| '''Grain'''          | 1 row per recipient event (Send / Open / Click / Bounce) with full HR context |
| '''Primary key'''    | Id (+ possibly composite — to be clarified via Q30) |
| '''Cross-channel key''' | TrackingId (inherited via join on Mailing master) |
| '''Refresh'''        | '''2x/day @ ~00:23 and ~12:25 UTC''' ('''CTAS Full Rebuild''' — no incrementality, Service Principal) — Q28 |
| '''Rows'''           | '''~520M''' (Jun 2021 - Apr 2026) |
| '''PII'''            | TNumber, possibly Receiver -> directly identifying |
| '''⚠️ Label confusion''' | Image in Q28 labelled the same table as tbl_pbi_platform_mailings — likely an image error (Q27 shows tbl_pbi_platform_mailings at 73K rows). Verify via Q30. |

!2 Primary Joins

!3 Pattern A: Direct consumption (default for dashboards)

{{{
SELECT TrackingId, Region, Division, LogStatus, COUNT(*) AS events
FROM   imep_gold.final
WHERE  event_time >= '2025-01-01'
  AND  TrackingId IS NOT NULL
GROUP BY TrackingId, Region, Division, LogStatus
}}}

!3 Pattern B: Pack-level aggregation (dashboard grain)

{{{
SELECT array_join(slice(split(UPPER(TrackingId), '-'), 1, 2), '-') AS tracking_pack_id,
       COUNT(DISTINCT CASE WHEN LogStatus = 'Sent'  THEN TNumber END) AS sent_unique,
       COUNT(DISTINCT CASE WHEN LogStatus = 'Open'  THEN TNumber END) AS open_unique,
       COUNT(DISTINCT CASE WHEN LogStatus = 'Click' THEN TNumber END) AS click_unique
FROM   imep_gold.final
WHERE  TrackingId IS NOT NULL
  AND  event_time >= '2025-01-01'
GROUP BY 1
}}}

!2 Quality Caveats

 * '''⚠️ CTAS Full Rebuild 2x/day''' — the table is completely destroyed and rebuilt. Long-running queries spanning the refresh windows (00:23 + 12:25 UTC) may encounter an inconsistent state.
 * '''No incrementality on the Gold side''' — any change in Bronze only lands in final after the next CTAS run. For out-of-band questions about newer events, use Bronze directly.
 * '''Full scan is expensive''': 520M rows × wide schema. Always restrict queries with event_time or TrackingId filter. According to Q28 the Gold CTAS has '''no Z-Order''' — no queries optimized for specific columns.
 * '''HR snapshot at build time''': Region/Division reflect the HR state at the time of the CTAS. If an employee moves EMEA -> APAC on July 1, all older final events show the new value '''after the next CTAS''' — not the historical one. For true temporal HR analyses, join from Bronze with a tbl_hr_employee snapshot.
 * '''Biggest compute hotspot in the pipeline budget''' — CTAS rebuild of a 520M-row table 2x/day. Identified as the first optimization lever (Q28), but not our concern.

!2 Consumption Strategy

 * '''Default''': use imep_gold.final. Pre-assembled, HR-ready.
 * '''If you can't find something in final''' (e.g. a specific Bronze column): don't try to extend final, join Bronze directly.
 * '''If final gets too large''': use the Tier-3 aggregates (tbl_pbi_mailings_region, _division, tbl_pbi_kpi) — they are pre-aggregated via GROUP BY MailingId × Dimension.
