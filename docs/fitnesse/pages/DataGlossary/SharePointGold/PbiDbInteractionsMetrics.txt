!1 PbiDbInteractionsMetrics ('''sharepoint_gold.pbi_db_interactions_metrics''')

'''Master interaction fact for SharePoint.''' 84M rows, 11 columns. Central table for PageView, Visit, Duration, and Comment metrics at Page × Date × Contact grain. '''Carries no direct TrackingID''' — cross-channel attribution always runs via marketingPageId -> pageUUID -> UBSGICTrackingID through sharepoint_bronze.pages. (Q22)

!2 Meta
| '''Layer'''          | Gold (Consumption) |
| '''Source system'''  | SharePoint Analytics (aggregated from sharepoint_silver) |
| '''Grain'''          | 1 row per marketingPageId × visitdatekey × viewingcontactid × referenceapplicationid |
| '''Primary key'''    | Composite (likely — to be clarified via Q30) |
| '''Cross-channel key''' | '''none directly''' — only via FK chain marketingPageId -> pageUUID -> UBSGICTrackingID |
| '''Refresh'''        | Daily Gold build (exact cadence — Q28 follow-up) |
| '''Rows'''           | '''~84M''' (from 2023) |
| '''PII'''            | viewingcontactid (GUID) -> indirectly identifying |

!2 Primary Joins

!3 Join to sharepoint_bronze.pages (N:1) — Standard lookup

{{{
SELECT m.*, p.UBSGICTrackingID, p.PageURL, p.SiteName
FROM   sharepoint_gold.pbi_db_interactions_metrics m
LEFT JOIN sharepoint_bronze.pages p ON p.pageUUID = m.marketingPageId
}}}

'''LEFT JOIN''', otherwise you lose the ~96% of rows without a populated TrackingID.

!3 Pack-level cross-channel funnel

{{{
SELECT array_join(slice(split(UPPER(p.UBSGICTrackingID), '-'), 1, 2), '-') AS tracking_pack_id,
       SUM(m.views)                  AS total_views,
       COUNT(DISTINCT m.viewingcontactid) AS unique_viewers,
       SUM(m.durationsum)            AS total_duration_sec
FROM   sharepoint_gold.pbi_db_interactions_metrics m
JOIN   sharepoint_bronze.pages p ON p.pageUUID = m.marketingPageId
WHERE  p.UBSGICTrackingID IS NOT NULL          -- mandatory filter
  AND  m.visitdatekey >= '20250101'            -- default time window from 2025
GROUP BY 1
}}}

!2 Quality Caveats

 * '''⚠️ No direct TrackingID''' — anyone searching with WHERE TrackingId = … will not find it. Always go via pages.UBSGICTrackingID.
 * '''4% coverage blocker''' (critical!): only 1,949/48,419 pages have UBSGICTrackingID. Of 84M interaction rows, '''only ~3.3M are Pack-attributable''' (4%). The remaining 96% are "untracked intranet activity". The dashboard '''must''' explicitly label this section.
 * '''viewingcontactid != TNumber''' — SharePoint-native person ID (GUID). Person-level attribution goes through pbi_db_employeecontact (still to validate).
 * '''Column typo''': commentss (with double s). Do not correct it or every query breaks.
 * '''marketingPageIdGuid''' is always NULL — ignore.
 * '''Grain anomaly''': a page × date × contact can theoretically have multiple rows (through referenceapplicationid variations) — for page-date grain rollup use GROUP BY marketingPageId, visitdatekey + SUM(views).
 * '''visitdatekey is a string''' — for time-range filters parse with CAST(visitdatekey AS date) or string compare visitdatekey >= '20250101'.

!2 Sister Tables in the Same Schema

For different grain requirements there are specialized Gold tables:

| '''Table'''                            | '''Rows''' | '''Cols''' | '''When to use''' |
| '''pbi_db_interactions_metrics'''      | 84M        | 11         | '''Default''' — everything in one table |
| pbi_db_pageviewed_metric               | 84M        | 5          | View counts only, fast aggregation |
| pbi_db_pagevisited_metric              | 81M        | 9          | Visit-oriented (with dedup) |
| pbi_db_datewise_overview_fact_tbl      | 7.5M       | 31         | Pre-aggregated page × date × division, rolling windows |
| pbi_db_90_days_interactions_metric     | 9M         | 11         | 90-day window, smaller, faster |

Rule of thumb: ad-hoc -> interactions_metrics; dashboards with time rollups -> datewise_overview_fact_tbl; pure count queries -> pageviewed_metric.
