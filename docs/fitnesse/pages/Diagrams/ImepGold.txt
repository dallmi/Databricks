!1 ER Diagram — iMEP Gold

Gold topology for iMEP. '''Strict 4-tier hierarchy''' (31 tables classified). All tables join via MailingId = tbl_pbi_platform_mailings.Id = tbl_email.Id back to the master. final is the sole Tier-0 atomic fact and carries 520M rows.

!2 4-Tier Hierarchy

| '''Tier''' | '''Role''' | '''# Tables''' | '''Representative members''' |
| Tier 0 — Atomic Fact | Per-Recipient × Event × Hour | 1 | final (520M, 64K mailings) |
| Tier 1 — Timespan × Dim Aggregates | 24h / 72h / 1w / 15w / 15m+ × Date/DateHour/DivArea/RegCntry | 15 | dateonly, datetime, divarea, regcntry variants |
| Tier 2 — Per-Mailing Summaries | UniqueOpens/UniqueClicks per Mailing × Dim | ~8 | mailingreceiver_*, engagement (1.8M, 117K mailings — '''mixed mail+event'''!), log_mail |
| Tier 3 — Platform & Reference Dims | Master + lookup tables | ~7 | tbl_pbi_platform_mailings (73,920 / 927 TIDs), tbl_pbi_platform_events (84,052) |

!2 Two Parallel Data Paths

!3 Path 1: Master-Detail (Tier 1 → Tier 2/3)
{{{
tbl_email (Bronze)
    |  [1:1 Full Rebuild]
    v
tbl_pbi_platform_mailings (Tier 1, 73K)
    |
    +--> tbl_pbi_mailings_region    (aggregated × Region)
    +--> tbl_pbi_mailings_division  (aggregated × Division)
    +--> tbl_pbi_kpi                (pivoted measures)
    +--> tbl_pbi_analytics
}}}
'''When to use''': aggregated metrics per Mailing × Dimension. Smaller, faster than final.

!3 Path 2: Denormalized Event Stream (final)
{{{
tbl_email                           -+
tbl_email_receiver_status           -|--[Full Rebuild]--> final (520M)
tbl_analytics_link                  -|
tbl_hr_employee + tbl_hr_costcenter -+
}}}
'''When to use''': event-level queries with HR context, without manual joining. Default for dashboards.

!2 The Pivoted tbl_pbi_kpi — Caution when Aggregating

tbl_pbi_kpi is '''not''' wide format. Instead:

| '''MailingId''' | '''measure''' | '''MailingReceiverStatus''' | '''value''' |
| a1b2c3…         | OpenCount     | Open                         | 1523        |
| a1b2c3…         | ClickCount    | Click                        | 189         |
| a1b2c3…         | SentCount     | Sent                         | 8500        |

For dashboard consumption pivot it:

{{{
SELECT MailingId,
       MAX(CASE WHEN measure = 'SentCount'  THEN value END) AS sent,
       MAX(CASE WHEN measure = 'OpenCount'  THEN value END) AS opened,
       MAX(CASE WHEN measure = 'ClickCount' THEN value END) AS clicked
FROM   imep_gold.tbl_pbi_kpi
GROUP BY MailingId
}}}

!2 NULL Semantics in Tier-3 Aggregates

In tbl_pbi_mailings_region and _division:
 * UniqueOpens NULL ~28-35% — mailings without a single open event in the region. NOT a data error.
 * UniqueClicks NULL ~66-81% — mailings without clicks in the region. Click rate is naturally lower than open rate.
 * Match rate Mailing ↔ Tier-3: 72-98% — not all mailings have Tier-3 entries at all.

Semantic NULLs, not defects. Use COALESCE(value, 0) for aggregations.

!2 Physical Storage

 * All 31 gold tables live in the '''shared gold ADLS account''' alongside sharepoint_gold.* — cross-channel joins inside Fabric/Spark possible without cross-account auth
 * Path pattern: abfss://gold@<gold-acc>/.../final or /tbl_pbi/* (+ 1 outlier /imep/tbl_active_employee_month)
 * External Delta, produced by a '''single orchestration notebook''' (Spark 3.2.1) — no managed pipeline
 * '''⚠️ Zero Partitioning''' on any table — biggest structural performance gap in the entire system; full scans on any query without tight filter

!2 Engagement-Table Anomaly

engagement (Tier 2, 1.8M rows) carries '''117,185 distinct mailingIds''', while tbl_pbi_platform_mailings only has 73,930. The delta (~44K) = event-IDs from tbl_pbi_platform_events. iMEP Gold treats '''emails + events as the same engagement unit''' in Tier-2 aggregates. Filter explicitly on the 73K mailing-IDs for mailing-specific analyses.
