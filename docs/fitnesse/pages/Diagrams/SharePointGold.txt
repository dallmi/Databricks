!1 ER Diagram — SharePoint Gold

Gold topology for SharePoint. '''Strict 4-tier hierarchy''' (20 tables + 3 Video sub-domain + 2 views). The FK chain runs via marketingPageId — all Gold metric tables reference back to sharepoint_bronze.pages.pageUUID. '''No direct TrackingID on the Gold facts!''' Cross-channel attribution always goes via pages.

!2 4-Tier Hierarchy

| '''Tier''' | '''Role''' | '''# Tables''' | '''Representative members''' |
| Tier 0 — Atomic Interaction Facts | Per-Visitor × Page × Date | 4 | pbi_db_interactions_metrics (84M), pageviewed_metric (84M), pagevisited_metric (81M), 90_days_interactions_metric (9M) |
| Tier 1 — Pre-Aggregated Overview | Rolling Windows (7/14/21/28d) over Page × Date × Division | 3 | datewise_overview_fact_tbl (7.5M, 27 cols) |
| Tier 2 — Engagement / Social Signals | Likes, Comments, CTA | 3 | pageliked_metric (400K), pagecommented_metric (500K), page_like_int_fact (30K) |
| Tier 3 — Dimension Tables | Person + Page + Calendar + Referrer | ~7 | employeecontact (24.3M, candidate person-bridge), website_page_inventory, calendar, referer_application |

Plus '''Video Sub-Domain (3 tables)''' — the only tables in the entire system produced by a managed Declarative Pipeline (Spark 3.5.2/4.0.0, ChangeDataFeed, proper comments). The rest is Notebook-basierter Full Rebuild (Spark 3.2.1)..

Plus '''2 Views''' (no physical storage): pbi_db_page_visitedkey_view, pbi_db_website_page_view.

!2 Three Critical Rules

!3 Rule 1 — Gold metrics without a pages JOIN are unattributed

{{{
-- WRONG — only raw numbers without cross-channel context
SELECT SUM(views) FROM sharepoint_gold.pbi_db_interactions_metrics;

-- RIGHT — attribution via TrackingID
SELECT p.UBSGICTrackingID, SUM(m.views) AS views
FROM   sharepoint_gold.pbi_db_interactions_metrics m
JOIN   sharepoint_bronze.pages p ON p.pageUUID = m.marketingPageId
WHERE  p.UBSGICTrackingID IS NOT NULL
GROUP BY p.UBSGICTrackingID
}}}

!3 Rule 2 — ~96% of rows are "untracked"

Only 4% of pages have a TrackingID. Of 84M interaction rows, '''~80M are unattributable'''. The dashboard must make this explicit — e.g. two separate sections.

!3 Rule 3 — viewingcontactid ≠ TNumber

SharePoint-native person ID is a GUID (viewingcontactid), not TNumber. For cross-channel joins with iMEP person data either:
 * pbi_db_employeecontact as bridge (hypothesis, not yet validated)
 * or fall back to sharepoint_bronze.pageviews.user_gpn + iMEP HR bridge

!2 All Paths Lead Through pages

'''None of the Gold metric tables carry GICTrackingID or UBSGICTrackingID directly.''' Deliberate normalization — the TrackingID lives only on the dimension.

!2 Physical Storage

'''Two pipeline families''' visible from the ADLS paths:

 * '''Family 1 "Employee Analytics" (16 tables)''' — abfss://gold@<gold-acc>/.../employee_analytics/pbi_db_*, Spark 3.2.1, Notebook-basierter Full Rebuild, no pipeline metadata
 * '''Family 2 "Video Analytics" (3 tables)''' — abfss://gold@<gold-acc>/.../sharepoint_gold/sharepoint_gold/pbi_db_*, Spark 3.5.2/4.0.0, managed Declarative Pipeline (pipelines.pipelineId), ChangeDataFeed enabled, proper table comments

'''Gold co-location with iMEP Gold''': both schemas live in the same ADLS gold account → cross-channel joins inside Fabric/Spark without cross-account auth.

'''⚠️ Zero partitioning''' on any of the 20 tables — interactions_metrics (84M) + pageviewed_metric (84M) trigger full scans on any query without a date filter.
