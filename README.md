# Multi-Channel Communication Analytics

Processes raw AppInsights `pageViews` telemetry into a CDM-based star schema
and — via the **CammsTrackingID** — joins it with email send/open/click data
from iMEP and event registrations, producing a unified cross-channel view of
every communication activity planned in CPLAN.

## Pipeline Overview

```
AppInsights Export (CSV/XLSX)
    |
    v
[1] Flatten customDimensions JSON
[2] Convert timestamps UTC -> CET
[3] Normalize GPN (8-digit, zero-padded)
[4] Temporal HR join via GPN (monthly snapshot matching)
[5] Compute time-on-page (next-view delta, 30min cap)
[6] Build star schema + session aggregation
    |
    v
pageviews.duckdb  +  Parquet files  +  Excel summary
```

## Usage

```bash
# Place AppInsights exports in input/, then run:

# Process a specific file:

python scripts/flatten_appinsights.py --full-refresh

python scripts/flatten_appinsights.py --hr /path/to/hr_history.parquet
```

## Data Model

### fact_page_view (1 row per page view)

| Column | Description |
|---|---|
| view_id | Primary key (from AppInsights) |
| timestamp | Event time (CET) |
| page_id | FK to dim_page |
| user_id | Anonymous browser ID (for unique visitor counts) |
| session_id | Session grouping |
| page_load_ms | Page load time (performance metric) |
| time_on_page_sec | Time until next view in session (engagement metric) |
| is_last_in_session | True = last page, time_on_page not measurable |
| referrer_url | Where the user came from |
| client_os / client_browser / client_country | Client info |
| gpn / email | Visitor identity (temporary, for HR join validation) |
| hr_division / hr_unit / hr_area / hr_sector | Organizational dimensions from HR |
| hr_region / hr_country | Work location from HR |
| hr_job_title / hr_management_level | Role info from HR |
| source_file | Input filename (for upsert tracking) |

### agg_session (1 row per session, derived from fact_page_view)

| Column | Description |
|---|---|
| session_id | Primary key |
| session_date / session_start / session_end | Timing |
| duration_sec | Last view minus first view |
| engagement_time_sec | Sum of measurable time-on-page |
| avg_time_on_page_sec | Average per page (excluding last) |
| page_view_count | Pages visited |
| entry_page_id / exit_page_id | First and last page |
| is_bounce | True if only 1 page view |
| hr_* | HR dimensions (from first view in session) |

### dim_page (deduplicated page metadata)

page_id, page_name, page_url, site_id, site_name, content_owner, content_type, theme, topic, target_region, target_org, page_status, publishing_date

### dim_date (CET-based date dimension)

date_key (YYYYMMDD), date, year, quarter, month, month_name, week, day_of_week

## Key Metrics

| Metric | How to compute |
|---|---|
| Views | `COUNT(*)` from fact_page_view |
| Unique Visitors | `COUNT(DISTINCT user_id)` from fact_page_view |
| Sessions / Visits | `COUNT(*)` from agg_session |
| Bounce Rate | `AVG(is_bounce)` from agg_session |
| Avg Pages/Session | `AVG(page_view_count)` from agg_session |
| Avg Time on Page | `AVG(time_on_page_sec)` from fact_page_view |
| Avg Session Duration | `AVG(engagement_time_sec)` from agg_session |

## Delta Load & Upsert

The script uses SHA-256 file hashes to detect changes:

- **New file**: rows are inserted
- **Unchanged file** (same hash): skipped
- **Changed file** (same name, different hash): old rows deleted, new rows inserted

A `processed_files` manifest table in DuckDB tracks filename, hash, row count, and processing timestamp.

## HR Join

Each page view is joined with the HR snapshot closest to the event month:

1. Find the most recent HR snapshot **at or before** the event month
2. Fallback: use the earliest snapshot **after** the event month
3. Join key: GPN (8-digit, normalized)

HR data source: `hr_history.parquet` (generated by `process_hr_history.py` in SearchAnalytics).

## CammsTrackingID — the cross-channel join key

Every communication activity planned in CPlan receives a system-generated
32-character tracking ID. It is emitted into every channel that delivers
the activity (Email, Intranet page, Event, Banner, ...). All delivery
telemetry carries the same ID, which is how we can reconstruct the full
user journey across channels.

### Format

```
QRREP-0000058-240709-0000060-EMI
  |       |       |       |     |
  |       |       |       |     +-- Channel abbreviation  (EMI=Email, INT=Intranet, EVT=Event, ...)
  |       |       |       +-------- Activity sequence number
  |       |       +---------------- Publication date (YYMMDD)
  |       +------------------------ Communication pack sequence number
  +-------------------------------- Tracking cluster (campaign program)
  |_______________|
   Communication pack ID (= cluster + pack number)
```

| Segment | Column | Example |
|---|---|---|
| 1 | `tracking_cluster_id` | `QRREP` |
| 2 | `tracking_pack_number` | `0000058` |
| 3 | `tracking_pub_date` | `240709` |
| 4 | `tracking_activity_number` | `0000060` |
| 5 | `tracking_channel_abbr` | `EMI` |
| 1+2 | `tracking_pack_id` | `QRREP-0000058` |

`flatten_appinsights.py` splits the raw ID into these 7 columns so every
segment can be filtered or joined independently.

### Channel mapping

| Abbr | Channel | Source system | Fact table in this repo |
|---|---|---|---|
| `EMI` | Email | iMEP (`imep_bronze.tbl_email_receiver_status`) | `fact_email` |
| `INT` | Intranet News article | AppInsights `pageViews` | `fact_page_view` |
| `EVT` | Event | iMEP (`imep_bronze.tbl_event_registration`) | `fact_event` |
| `BAN` | Banner | AppInsights (banner impressions) | (future) |
| `SMS` | SMS | (future) | (future) |

### Cross-channel join model

```
                        ┌─ CPLAN ────────────────┐
                        │  packs (cluster+pack#) │
                        │  activities (channel)  │
                        └───────────┬────────────┘
                                    │  tracking_pack_id
                                    │  tracking_id
             ┌──────────────────────┼──────────────────────┐
             │                      │                      │
     ┌───────┴────────┐    ┌────────┴────────┐    ┌────────┴────────┐
     │  Email (iMEP)  │    │ Intranet (AI)   │    │ Events (iMEP)   │
     │  sent/open/clk │    │ page views      │    │ registrations   │
     └────────────────┘    └─────────────────┘    └─────────────────┘
```

**Grain**:

- **Activity level** (`tracking_id`): joins a single email send to a single
  intranet page to a single event — 1 tracked URL per activity.
- **Pack level** (`tracking_pack_id`): rolls activities of the same
  communication pack up across channels — usually what the dashboard
  shows (one row per campaign).
- **Cluster level** (`tracking_cluster_id`): aggregates entire campaign
  programs.

### Deriving a cross-channel fact

```sql
-- One row per pack, columns per channel
WITH email AS (
  SELECT tracking_pack_id,
         COUNT(*) FILTER (WHERE status = 'sent')    AS sent,
         COUNT(*) FILTER (WHERE status = 'opened')  AS opened,
         COUNT(*) FILTER (WHERE status = 'clicked') AS clicked
  FROM fact_email GROUP BY 1
),
intranet AS (
  SELECT tracking_pack_id,
         COUNT(*)                 AS views,
         COUNT(DISTINCT user_id)  AS unique_readers,
         AVG(time_on_page_sec)    AS avg_tos
  FROM fact_page_view GROUP BY 1
),
events AS (
  SELECT tracking_pack_id,
         COUNT(*) FILTER (WHERE status = 'registered') AS registered,
         COUNT(*) FILTER (WHERE status = 'attended')   AS attended
  FROM fact_event GROUP BY 1
)
SELECT COALESCE(e.tracking_pack_id, i.tracking_pack_id, v.tracking_pack_id) AS pack,
       e.sent, e.opened, e.clicked,
       i.views, i.unique_readers, i.avg_tos,
       v.registered, v.attended
FROM email e
FULL JOIN intranet i USING (tracking_pack_id)
FULL JOIN events   v USING (tracking_pack_id);
```

### Funnel across channels

```
Email sent ──► Email opened ──► Email link clicked ──► Intranet page view ──► Event registered
   100%            45%                8%                     6%                       1.5%
```

The dashboard (see [`dashboard/multi_channel.html`](dashboard/multi_channel.html))
computes this funnel per pack, per cluster and per date range. Seed data is
baked into the HTML so the dashboard runs out of the box without Databricks
connectivity — replace the seed JSON with exports from
`fact_email` / `fact_page_view` / `fact_event` once the pipeline is wired up.

## Ad-hoc Queries

```bash
# Open DuckDB CLI
duckdb output/pageviews.duckdb

# Top 10 pages by unique visitors
SELECT p.page_name, p.site_name, COUNT(DISTINCT f.user_id) as uv
FROM fact_page_view f JOIN dim_page p ON f.page_id = p.page_id
GROUP BY p.page_name, p.site_name ORDER BY uv DESC LIMIT 10;

# Bounce rate by division
SELECT hr_division, COUNT(*) as sessions,
       ROUND(AVG(is_bounce::INT) * 100, 1) as bounce_pct
FROM agg_session GROUP BY hr_division ORDER BY sessions DESC;

# Weekly trend
SELECT d.year, d.week, COUNT(*) as views, COUNT(DISTINCT f.user_id) as uv
FROM fact_page_view f JOIN dim_date d ON CAST(f.timestamp AS DATE) = d.date
GROUP BY d.year, d.week ORDER BY d.year, d.week;
```
